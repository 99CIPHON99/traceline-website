Using Traceline

Requirements & Issues

Requirements traceability answers "what's actually covered by testing?" Issues tracking answers "what did we find, and did we fix it?" Traceline links both back to your test cases.

Requirements traceability

Add a requirement (Quality → Requirements) with a reference code, title, and description, then link it to the test cases that verify it via Cases on that requirement.

Each requirement shows a live coverage badge based on the latest result of every linked test case:

BadgeMeaning
Not coveredNo test cases linked yet
N linked · all passingEvery linked case's most recent result was Passed
N linked · N failingAt least one linked case's most recent result was Failed or Blocked
N linked · N passedPartial coverage — some passed, none failing yet

You can also generate draft test cases directly from a requirement via AI Cases — see AI Features.

Issues (kanban board)

Issues → board view shows a four-column kanban: Open, In Progress, Resolved, Closed. Drag a card between columns to change its status. Each card shows its priority (color-coded left edge), type, project, and assignee.

Switch to table view for a sortable list instead — useful when you need to filter by status/priority/type rather than drag cards around.

Issues can be created directly, or drafted automatically from a failed test result — see Test Execution.

Reopened issues

An issue moved back to Open after having been Resolved or Closed shows a REOPENED badge on its card, so regressions don't quietly blend in with new bugs.

Jira integration

Once Jira is connected (Settings → Integration Settings → Jira), any issue without a linked Jira key shows a Push to Jira button on its kanban card. Pushing creates the Jira issue and stores its key on the Traceline issue — from then on the card shows a linked-Jira badge instead.

Configure the connection with your Jira base URL, account email, and an API token (Atlassian → Account Settings → Security → API tokens) — Traceline encrypts this credential at rest.
