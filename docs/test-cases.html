Using Traceline

Test Cases

Test cases are the reusable heart of Traceline. This page covers creating them by hand, importing an existing library via CSV, how automatic classification works, and versioning.

Creating a test case

From a Suite or Section, click New Test Case. Each case has:

  • Title — required, shown everywhere the case is referenced
  • Preconditions — state the system must be in before the steps run
  • Steps — an ordered list of step + expected-result pairs
  • Expected Result — the overall outcome
  • Priority — low, medium, high, critical
  • Type — functional, acceptance, performance, regression, smoke, integration
  • Automation status — not automated, in progress, automated, not applicable
  • Tags — free-form, comma-separated

Importing from CSV

Test Cases → Import CSV. The importer auto-detects a couple of common export formats. The only required column is:

ColumnRequiredNotes
titleYesCase-insensitive header match
section or modulenameNoUsed as a fallback if step-based classification (below) doesn't find a clearer signal
case code / casecodeNoPreserves an existing ID from your old tool
priorities / priorityNoDefaults to Medium if omitted
typeNoDefaults to Functional if omitted

Column headers are matched case-insensitively with whitespace/quotes trimmed, so most real-world exports work without pre-processing.

Automatic classification

If your test steps contain navigation breadcrumbs in the form GOTO <System>., Traceline uses the first GOTO target as the primary signal for which Suite and Section a case belongs to — this is more reliable than a flat "module name" column when the same module name is reused across multiple products.

Step 1: GOTO Administration. GOTO Users. Click "Add User".
Expected: New user form appears.

This example would classify into the Administration suite, Users section — regardless of what a "ModuleName" column in the CSV says.

Already imported cases with the wrong classification? Test Cases → Reclassify Modules re-runs classification on every existing case using this same logic and moves anything that's filed under the wrong suite/section. Safe to run anytime — correctly-classified cases are left alone.

Shared "Administration" suite

Administrative test cases (user management, permissions, system settings) that apply across every product get pulled into a shared Administration (All Products) suite automatically, grouped by which product they came from underneath it — while still remaining accessible from within each product's own suite too. You're not maintaining two copies.

Versioning

Every edit to a test case's title, preconditions, steps, expected result, priority, or tags creates a new version behind the scenes. Open a test case → History to see prior versions and who changed what.

Shared Steps

For a sequence you reuse across many cases (e.g. "Log in as admin"), create it once under Shared Steps and reference it from any test case instead of retyping it — editing the shared step updates everywhere it's referenced.

Bulk operations

Select multiple test cases via their checkboxes to bulk-delete, or delete an entire suite's cases at once from Test Cases → module actions.
