Getting Started

Core Concepts

Five ideas explain almost everything in Traceline. Understand these and the rest of the app is just where the buttons are.

The hierarchy

Test cases live in a strict hierarchy — this is what makes large test libraries navigable instead of one giant flat list:

Project
└── Suite            (e.g. a product, module, or major feature area)
    └── Section       (a folder within a suite — can be nested)
        └── Test Case (the actual reusable test)
  • Project — the top-level container. Most teams run one project per product; some run one per team.
  • Suite — a grouping within a project. A project can have suite_mode set to single, multiple, or versioned suites depending on how you organize things.
  • Section — a folder inside a suite for further organization (e.g. "Login", "Checkout", "Admin Panel").
  • Test Case — the reusable, versioned unit: title, preconditions, steps, expected result, priority, type, and automation status.

Test cases are not tied to a single execution. The same case gets reused across every Test Run — that's what makes a "did this pass last time" history possible.

Test Cases vs. Test Runs

This is the concept that trips up people coming from a simple bug tracker:

  • A Test Case is the reusable definition — "Login with valid credentials succeeds." It doesn't have a pass/fail state by itself.
  • A Test Run is a specific execution pass — "Sprint 42 Regression" — that snapshots a set of test cases and tracks pass/fail/blocked/retest/skipped for each one, on that run, at that time.

This is why the same test case can show "passed" in one run and "failed" in another — the result belongs to the run, not the case.

Roles

Every user has exactly one role, which determines what they can do:

RoleCan do
adminEverything — user management, all CRUD, delete, settings, licensing
leadAll test management, plus delete suites and runs
qaCreate and edit test cases and results
developerView and execute test runs
viewerRead-only access everywhere
