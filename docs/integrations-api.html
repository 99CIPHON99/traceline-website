Extending Traceline

Integrations & API

Slack notifications, generic outbound webhooks, a token-authenticated REST API for CI pipelines, and an MCP client for connecting external tool servers.

Slack notifications

Settings → Integration Settings → Slack. Paste an Incoming Webhook URL from your Slack workspace. Once connected, events like new issues are posted automatically. Delivery happens asynchronously — creating an issue never waits on Slack's response time.

Outbound webhooks

Settings → Integration Settings → Webhooks. Add any HTTPS endpoint, choose which events it should receive (or subscribe to all with *), and optionally set a signing secret.

Each delivery includes an HMAC-SHA256 signature header when a secret is configured:

X-Traceline-Signature: <hmac_sha256(secret, request_body)>

Payload shape:

{
  "event": "issue_created",
  "title": "New issue: ISS-042",
  "message": "Login button unresponsive on retry",
  "context": { "url": "https://your-install/issues", "priority": "high" },
  "sent_at": "2026-07-05T14:22:00+00:00"
}

Use the Test button next to any webhook to send a one-off test payload regardless of its event subscriptions.

REST API for CI pipelines

Generate a token: Settings → Integration Settings → API Tokens → Generate. The plaintext token is shown exactly once — store it in your CI secrets immediately.

Push a result

curl -X POST https://your-install/api/v1/results \
  -H "Authorization: Bearer ttk_xxxxxxxxxxxxxxxxxxxx" \
  -H "Content-Type: application/json" \
  -d '{
    "run_id": 12,
    "case_code": "TC0001",
    "status": "passed",
    "comment": "optional"
  }'

Identify the test case with either case_code or test_case_id. Valid status values: passed, failed, blocked, retest, skipped, untested. Calling this again for the same run+case updates the existing result rather than duplicating it.

Check a run exists

curl https://your-install/api/v1/runs/12 \
  -H "Authorization: Bearer ttk_xxxxxxxxxxxxxxxxxxxx"

Useful as a pre-flight check in a CI script before pushing a batch of results.

MCP client

Settings → Integration Settings supports connecting to Model Context Protocol servers (Streamable HTTP transport, spec 2025-03-26). Once connected, Browse Tools lists what the server exposes and lets you invoke them directly from Settings — useful for testing an MCP server integration without a separate client.

Legacy SSE-only and OAuth-authenticated MCP servers aren't supported yet — only Streamable HTTP with bearer-token auth.
